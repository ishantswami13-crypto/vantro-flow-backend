## 1. Executive Summary

Atlas by Vantro is a global AI Business Automation OS — not a chatbot, not a dashboard, not a reporting tool. It is a specialized automation system where every repeatable business decision is encoded as a scoped, governed, audited agent.

**Public claim (approved):** "12 core specialized agents with an expandable Agent Mesh architecture."

**Internal design:** 216 specialized agents across 6 layers, covering every domain from CashOps and collections to enterprise governance, infrastructure reliability, and GTM operations.

**Critical rule:** 216 is internal architecture, registry, and proof-gated system design. The number "216" does not appear on the public website until all proof gates are met.

### What Atlas Is Not
- Not a generic AI assistant or chatbot
- Not a financial reporting dashboard
- Not a CRM with AI features bolted on
- Not an autonomous system that acts without owner approval

### What Atlas Is
- A specialized AI automation OS for business operations
- Every agent has a defined scope, defined tools, defined outputs, defined policy guards
- Every critical action requires human approval
- Every execution is audited, cost-tracked, and Harness X verified

## 3. Core Architecture Principles

1. **React-first operator experience** — Every operator interaction goes through React. No raw API calls, no CLI. React is the operator's control plane.

2. **Cortex Core RS (Rust) orchestration** — The deterministic engine. Routing, policy enforcement, parallel execution, cost tracking — all in Rust. Zero undefined behavior.

3. **Parallel execution wherever safe** — Read-only agents run in parallel by default. Write agents use dependency graphs to determine safe concurrency.

4. **Queue-based execution for long-running workflows** — Redis + BullMQ for job queuing. No synchronous blocking on 30-second agent runs.

5. **Memory and retrieval for business context** — Every agent can access structured memory (PostgreSQL) and semantic memory (vector store) to understand the business's history.

6. **Intelligent LLM routing** — Claude Haiku for summaries, Sonnet for analysis, Opus for critical decisions. Cost-optimized per execution.

7. **Tool-driven agents** — Agents don't hallucinate actions. They call specific, registered tools. No tool = no action.

8. **Cost intelligence in every workflow** — Every token spent is tracked. Every agent has a cost budget. Circuit breakers prevent runaway costs.

9. **Harness X validates before production** — No agent runs in production without passing static, dry-run, and (for high/critical) red-team harness tests.

10. **Database-backed state and caching** — PostgreSQL is the source of truth. Redis accelerates reads. No ephemeral state that gets lost on restart.

### 4.2 Backend — Cortex Core RS and Supporting Services

#### Cortex Core RS (Rust)
The deterministic orchestration engine. Written in Rust for memory safety, zero-cost abstractions, and predictable performance.

Responsibilities:
- Receive execution requests from the Node.js API layer
- Load agent definitions from the registry
- Validate inputs against agent schemas
- Apply policy rules (policyGuard)
- Route to appropriate execution engine (direct, queue, scheduled)
- Coordinate parallel agent execution with dependency resolution
- Apply cost routing and LLM selection
- Invoke Harness X checks for high/critical agents in dry-run mode
- Emit structured audit events
- Return structured outputs to API layer

Current status: Built (`vantro-automation-rs` Axum sidecar on port 3002). Flag: `RUST_AUTOMATION_API_ENABLED=false`. Enable gate: cargo test + bootstrap <500ms + harness pass.

#### Agent Registry Engine
- Loads all production-status agents at startup
- Caches in Redis with 30-minute TTL
- Validates agent schemas on every load
- Exposes `GET /api/v1/agents` and agent-specific endpoints
- Enforces feature flag gating per agent

#### Workflow Orchestrator
- Manages multi-step agent workflows
- Tracks workflow state in PostgreSQL
- Handles step failures with configurable retry policies
- Supports workflow pause/resume (for approval gates)
- Emits workflow lifecycle events to audit pipeline

#### Parallel Execution Engine
- Analyzes agent dependency graphs
- Identifies agents that can run concurrently
- Uses Tokio async runtime for concurrent execution
- Aggregates results from parallel branches
- Handles partial failures gracefully

#### Queue & Background Job System
- Redis + BullMQ for job queuing
- Job priority levels: critical (immediate), high (1min), standard (5min), batch (hourly)
- Dead-letter queue for failed jobs
- Retry policies: exponential backoff with jitter
- Background job monitoring dashboard
- Handles: daily briefing generation, overnight risk recalculation, batch collection prioritization

#### Memory & Retrieval Engine
- Short-term: Redis (current session, 5-minute TTL)
- Long-term structured: PostgreSQL (business_memory table: customer patterns, payment history, behavior scores)
- Long-term semantic: pgvector (embedding-based retrieval for unstructured business context)
- Entity memory: Customer profiles, supplier relationships, invoice patterns
- Retrieval: Hybrid search (semantic similarity + structured query)

#### Cost Intelligence Engine
- Tracks every LLM call: model, tokens in/out, cost USD
- Enforces per-agent cost budgets
- Circuit breaker: agent suspended if cost > budget × 2 in 30 days
- Alerts at 75% and 90% of monthly budget
- Routes to cheaper models when quality is sufficient
- Reports cost per business outcome

#### Policy & Approval Engine
- Loads policy rules per agent
- Evaluates rules in priority order
- Actions: allow, deny, require_approval, flag_for_review, add_disclaimer
- Tenant-level policy overrides (enterprise)
- Creates approval requests for critical agents
- Enforces approval timeouts and escalations

#### Audit Event Pipeline
- Every agent execution emits structured audit events
- Events: agent.triggered, agent.executed, agent.proposed, approval.requested, approval.decided, action.executed
- Immutable append-only log in PostgreSQL
- Real-time streaming to Grafana
- Tenant-scoped: business A cannot see business B's events

#### Harness X Validation Layer
- Static analysis: schema validation, policy check, tool availability
- Dry-run execution: synthetic data, no side effects
- Red-team execution: adversarial inputs, boundary testing
- Live validation: staging environment with real data
- Regression guard: compares current vs. baseline results
- Gate enforcement: blocks production promotion on failure

- pgvector extension on PostgreSQL (development/staging)

#### Event-Driven Architecture
- EventEngine (lib/events/EventEngine.js) as the internal event bus

- Grafana for dashboards and alerting
- Prometheus metrics (prom-client)
- Structured logging (logger.js)
- Error tracking (error-tracking.js)

- All agents default to disabled (feature flag OFF)
- `lib/featureFlags.js` is the source of truth

## 7. Cortex Core RS Execution Flow

```
1. BUSINESS EVENT
   → Invoice marked overdue, payment received, daily cron, owner request
   
2. EVENT NORMALIZER (cortex.event_normalizer)
   → Standardizes event format, adds metadata, deduplicates
   
3. CONTEXT BUILDER (cortex.context_builder)
   → Assembles business context: tenant, user, relevant records
   
4. MEMORY RETRIEVAL
   → Loads customer history, payment patterns, previous agent outputs
   → Short-term from Redis, long-term from PostgreSQL + vector store
   
5. AGENT ROUTER (cortex.agent_router)
   → Determines which agents should respond to this event
   → Checks feature flags, agent status, business tier
   
6. PARALLEL SPECIALIST AGENT EXECUTION
   → Multiple agents run concurrently where dependencies allow
   → E.g., Collections Priority + Credit Risk + Tone Strategy in parallel
   
7. CORTEX CORE RS DETERMINISTIC ENGINE
   → Receives agent inputs, applies scoring algorithms
   → Simulation engine calculates outcome probabilities
   → Results are deterministic given the same inputs
   
8. POLICY GUARD (policyGuard.service.js)
   → Evaluates all policy rules for each agent output
   → Enforces collections ethics, financial accuracy, consent rules
   → Blocks or flags outputs that violate policy
   
9. COST ROUTER (core.cost_router)
   → Evaluates whether LLM is needed for this execution
   → Routes to Haiku (cheap), Sonnet (default), Opus (critical)
   → Applies prompt compression if token budget is tight
   
10. LLM ROUTING LAYER
    → Sends prompt to selected model with caching
    → Validates output against expected schema
    → Retries on schema failure with correction prompt
    
11. HARNESS X CHECK (if risk_level: high or critical)
    → Runs output through safety checks before proceeding
    → Blocks if unsafe message patterns detected
    → Checks for hallucinated financial numbers
    
12. OWNER / MANAGER APPROVAL (if approval_required: true)
    → Creates approval request in PostgreSQL
    → Sends notification to owner/manager via React Approval Center
    → Workflow pauses until approved or expired
    
13. QUEUE OR IMMEDIATE EXECUTION
    → Immediate: low/medium risk, <5 seconds expected
    → Queue: high/critical, long-running, batch operations
    
14. AUDIT TRAIL
    → Every event logged: agent triggered, executed, proposed, approved
    → Immutable, tenant-scoped, timestamped
    
15. OUTCOME TRACKING
    → Records output and expected outcome
    → Links to downstream events to measure success
    
16. LEARNING & MEMORY UPDATE (core.learning)
    → Updates business memory with new patterns
    → Adjusts customer scores based on new behavior data
    → Feeds outcome data back to scoring models
    
17. OWNER BRIEFING (core.owner_briefing)
    → Synthesizes all events into daily/real-time briefing
    → Surfaces what matters most
    → Owner sees actions, not data
```

## 8. Why 216 Agents Don't Create Chaos — Cortex as Control Plane

The agent count is not chaos. Cortex is the control plane that makes it work.

**Routing:** Cortex decides which agents respond to each event. An invoice.overdue event triggers Collections Agent + Credit Risk Agent. It does NOT trigger the Regional Localization Agent or the Deployment Readiness Agent. Every agent only activates on its registered trigger events.

**Permissions:** Every agent has a `react_permissions_scope`. A collections manager can see Collection Priority Agent outputs. They cannot see Security Agent outputs or Deployment Readiness Agent outputs.

**Orchestration:** The Workflow Orchestrator manages dependencies between agents. The Tone Strategy Agent can only run after the Collections Priority Agent produces a ranked list. The right order is always enforced.

**Memory:** The Memory & Retrieval Engine ensures agents have relevant context without overwhelming them. Customer history is fetched for the Collections Agent. Infrastructure logs are fetched for the Deployment Readiness Agent. Each agent gets exactly its context.

**Policy:** Every agent runs through the Policy Guard. Rules are enforced at the Cortex level, not the agent level. One policy change applies to all agents that reference that rule.

**Approvals:** The Approval Engine is a gate, not a bottleneck. Low/medium risk agents run automatically. High risk agents propose. Critical agents require explicit approval. The owner only sees decisions that matter.

**Cost:** The Cost Intelligence Engine tracks every execution. Agents that exceed budgets are paused automatically. The owner is not surprised by a ₹50,000 LLM bill.

**Caching:** Results are cached where appropriate. The Collections Priority list doesn't need recalculation every 30 seconds. Cache + TTL keeps costs down without sacrificing freshness.

**Queue execution:** Long-running agents run in the background. The owner doesn't wait 30 seconds for a cashflow forecast. It's generated in the background, ready when they open the app.

**Fallback behavior:** Every agent has a fallback. If the LLM is unavailable, the agent falls back to deterministic rules. If the deterministic rules fail, it surfaces raw data. The owner always gets something useful.

**Auditability:** Everything is logged. Every execution, every proposal, every approval. If something goes wrong, there is a complete chain of evidence.

**Observability:** Grafana dashboards show agent health in real time. An agent that starts failing will trigger an alert before the owner notices.

## 10. Backend Architecture Deep Dive

### 10.1 Rust Deterministic Engines

**Why Rust for Cortex:**
- Memory safety without garbage collection: no GC pauses during scoring calculations
- Zero-cost abstractions: safe high-level code compiles to efficient machine code
- Tokio async runtime: handles thousands of concurrent agent executions
- Type system enforces correctness: policy rules cannot be applied incorrectly at compile time
- Predictable performance: no JIT warm-up, consistent sub-millisecond scoring latency

**Current Rust services:**
- `cortex-core-rs`: scoring.rs, simulation.rs, policy.rs — CLI binary
- `vantro-automation-rs`: Axum HTTP sidecar on port 3002 — cashops, cortex, api modules

### 10.2 Cortex Orchestrator

Event ingestion → normalize → build context → route to agents → parallel execution → aggregate → policy check → output

The orchestrator is stateless: all state lives in PostgreSQL. An orchestrator restart doesn't lose workflow progress.

### 10.3 Parallel Execution Framework

Safe parallelism rules:
- **Always parallel:** read-only agents (risk_level: low, no tools that write)
- **Parallel with locks:** agents that read the same entity (customer) — reads are parallel, writes are serialized
- **Sequential only:** agents in an explicit dependency chain (A → B → C)
- **Never parallel:** critical agents — always sequential with explicit approval gate

### 10.4 Queue and Background Job Processing

| Priority | Examples | Max Wait |
|----------|---------|----------|
| Critical | Approval notifications, security alerts | Immediate |
| High | Collections priority generation, cashflow alerts | 60 seconds |
| Standard | Daily briefing generation, risk recalculation | 5 minutes |
| Batch | Historical pattern analysis, memory updates | 60 minutes |

Failed jobs: 3 retries with exponential backoff (1s, 4s, 9s), then dead-letter queue. Admin alert on DLQ accumulation.

### 10.5 Database and Caching Strategy

| Cache Layer | TTL | Contents |
|------------|-----|----------|
| Agent Registry | 30 min | All production agent definitions |
| Business Context | 5 min | Current tenant business state |
| Customer Scores | 10 min | Payment behavior scores |
| LLM Responses | 60 min | Cacheable LLM outputs (summaries, risk assessments) |
| Collections Priority | 5 min | Priority list per tenant |

### 10.6 LLM Routing and Model Selection

| Model | Use Case | Cost | Speed |
|-------|---------|------|-------|
| Claude Haiku | Summaries, low-risk analysis, templated outputs | ✓✓✓ | Fast |
| Claude Sonnet | Analysis, recommendations, medium-risk decisions | ✓✓ | Medium |
| Claude Opus | Critical decisions, legal wording review, complex reasoning | ✓ | Slow |
| No LLM | Deterministic scoring, rule evaluation, data aggregation | Free | Instant |

Routing logic (cost.model_selection agent):
1. Evaluate task complexity score (0-100)
2. Check risk level of agent
3. Check token budget remaining
4. Select: ≤30 complexity → Haiku; 31-70 → Sonnet; 71+ or critical → Opus

### 10.7 Tool Execution Framework

Tool categories:
- **Reader tools:** invoice_reader, customer_history_reader, cashflow_reader (read-only, always safe)
- **Calculator tools:** scoring_calculator, simulation_engine, risk_calculator (deterministic, no side effects)
- **Writer tools:** workflow_state_writer, memory_writer, audit_logger (require explicit permission)
- **Communication tools:** whatsapp_sender, email_sender (critical — require owner approval + FEATURE_EXTERNAL_MESSAGE_SENDING_ENABLED=true)
- **Retrieval tools:** vector_search, memory_search, entity_lookup (read-only, cached)

### 10.8 Cost Optimization Engine

Monthly cost lifecycle:
1. Agent executes → cost tracked in agent_costs table
2. Daily rollup → per-agent monthly total calculated
3. At 75% budget → alert sent to admin
4. At 90% budget → warning in React Cost Dashboard + owner notified
5. At 100% budget → circuit breaker: agent execution suspended
6. At 200% budget → permanent suspension until admin reviews

### 10.9 Memory and Retrieval Architecture

```
Short-term (Redis, 5-min TTL)
└── Current session context, active workflow state

Long-term Structured (PostgreSQL)
└── business_memory table: customer_id, pattern_type, pattern_data, confidence_score
└── Examples: "Customer ABC pays 15 days late on average", "Supplier XYZ delays 30% of deliveries"

Long-term Semantic (pgvector / Pinecone)
└── Embeddings of: customer notes, past interactions, business context
└── Retrieved by semantic similarity to current query

Entity Memory (PostgreSQL)
└── customer_profiles: scoring history, behavior patterns, relationship risk
└── supplier_profiles: reliability scores, payment pattern preferences
└── invoice_patterns: typical payment timing per customer
```

### 10.10 Harness X Validation Pipeline

```
Agent promotion request
    → Static Harness (schema, policy, tool availability)
    → Dry-Run Harness (synthetic data, no side effects)
    → [HIGH/CRITICAL] Red-Team Harness (adversarial inputs, injection tests)
    → [STAGING] Live Harness (real data, staging env)
    → [HIGH/CRITICAL] Performance Harness (load test, SLA verification)
    → All green → Production promotion allowed
```

## 13. Critical Agent Rules — Non-Negotiable

1. **Critical agents cannot execute directly.** They propose. The human approves. Then it executes.

2. **No irreversible action without double confirmation.** Deleting a record, sending a legal notice, marking a payment status — all require two-factor approval (agent proposes + owner explicitly approves).

3. **24-hour cooling period** for certain critical financial record changes (e.g., writing off a debt).

4. **Audit log is mandatory.** Not optional. Every critical agent proposal, every approval, every rejection, every execution — logged with user_id, timestamp, reason.

5. **Harness X is mandatory.** No critical agent reaches staging without passing red-team scenarios including adversarial input injection and cross-tenant isolation tests.

6. **Communication agents have a hard gate:** `FEATURE_EXTERNAL_MESSAGE_SENDING_ENABLED=true` AND owner approval for each send. Two gates, not one.

7. **Financial record changes have a truth guard:** The Payment Truth Guard Agent validates every proposed payment status change against bank data before it can proceed.
