# Rust SQLx Validation -- Process Guide

## Why sqlx::query! needs a database or offline cache

`sqlx::query!` is a **compile-time macro**. When the Rust compiler processes
`vantro-automation-rs`, sqlx performs a round-trip to a Postgres database to
fetch the types of every column referenced in every query. This gives you
type-safe query results, but it means:

- A `DATABASE_URL` pointing to a compatible schema **must** be available at
  build time, **OR**
- A committed `.sqlx/` offline cache directory must exist in the repo.

Without one of these two things, `cargo build --features server` fails with:

```
error: failed to find data for query ...
```

## Why we must NOT use the production DATABASE_URL at build time

Two reasons:

1. **Security.** Injecting production Supabase credentials into CI build
   infrastructure creates a credential-in-pipeline risk. A single workflow
   misconfiguration could expose the key in logs or to forked PRs.

2. **Architecture.** The Rust server binary should build *offline* -- Railway
   NIXPACKS does not have access to the production database during the build
   phase. Building with production credentials during development creates
   environment parity mismatch: "works on dev machine, fails on Railway."

The correct answer is **`SQLX_OFFLINE=true`** + a committed `.sqlx/` cache.

## The offline cache: what it is

`cargo sqlx prepare` generates a `.sqlx/` directory at the workspace root.
Inside are JSON files -- one per `sqlx::query!` invocation -- that record the
expected column names, types, and nullability. When `SQLX_OFFLINE=true`, the
compiler reads these JSON files instead of connecting to the database.

The `.sqlx/` directory must be committed to the repository and kept in sync
with the queries in `db/queries.rs`.

## How the CI validation works

The `.github/workflows/rust-sqlx-validation.yml` workflow:

1. Spins up an ephemeral `postgres:16` container with no persistent state.
2. Applies `db/sqlx-test-schema.sql` -- a minimal CI-only schema that defines
   exactly the tables and columns that `queries.rs` references.
3. Runs `cargo sqlx prepare --workspace -- --features server` against that DB.
4. Runs `cargo build --profile ci-release --features server` with
   `SQLX_OFFLINE=true` to prove the generated cache is complete.
5. Uploads `.sqlx/` as a CI artifact named `sqlx-offline-cache`.

No production credentials are used at any step.

## The test schema vs production schema

`db/sqlx-test-schema.sql` is **not** the production schema. It exists solely
to give `cargo sqlx prepare` valid type information. Key differences:

| | Production | CI test schema |
|---|---|---|
| Location | Supabase + migrations/001-006 | db/sqlx-test-schema.sql |
| RLS | Enabled (migration 006) | Disabled (not applicable) |
| FK cascades | Full referential integrity | Minimal FKs for sqlx |
| All columns | Yes | Only columns used by Rust queries |
| Indexes | Full production indexes | None (not needed for prepare) |
| Applied by | You (Supabase SQL Editor) | CI automatically |

If a column is added to a query in `queries.rs` but not to
`db/sqlx-test-schema.sql`, the CI job fails with a clear error identifying
the missing column. Add it to the test schema and re-run.

## Step-by-step: generating and committing .sqlx/ for the first time

**Precondition:** the `rust-sqlx-validation` CI workflow has run and passed
on your branch.

### Step 1 -- Trigger the CI workflow

Push or open a PR. The workflow runs automatically on every push.

### Step 2 -- Download the artifact

1. Open the GitHub Actions run for `Rust SQLx Validation`.
2. Scroll to **Artifacts** at the bottom of the summary page.
3. Download `sqlx-offline-cache.zip`.
4. Unzip it into the repository root as `.sqlx/`:
   ```bash
   unzip sqlx-offline-cache.zip -d .sqlx/
   ```

### Step 3 -- Inspect the generated files

```bash
ls .sqlx/
# Expected output: one .json file per sqlx::query! invocation
```

Each JSON file records the SQL text (hashed), column names, and types.
Briefly verify the file count matches the number of `query!` calls in
`queries.rs` (currently 9 macro invocations across 3 functions).

### Step 4 -- Commit the offline cache

```bash
git add .sqlx/
git commit -m "chore: add SQLx offline cache for Rust automation

Generated by the rust-sqlx-validation CI workflow against the
ephemeral Postgres test schema (db/sqlx-test-schema.sql).
No production database was used.

SQLX_OFFLINE=true is required for all builds after this commit."
```

### Step 5 -- Verify activation in rust-automation-ci.yml

After this commit, the `verify-server-feature-build` job in
`.github/workflows/rust-automation-ci.yml` will activate automatically -- it
is gated on `if: ${{ hashFiles('.sqlx/**') != '' }}`.

The next CI run on `rust-automation-ci.yml` should show all three jobs green:

```
✅ verify-pure-rust
✅ verify-cortex-harness-static
✅ verify-server-feature-build   ← now active
```

## Step-by-step: refreshing .sqlx/ after a query change

Any change to a `sqlx::query!` in `queries.rs` requires the `.sqlx/` cache
to be regenerated. The process:

1. Make the query change in `queries.rs`.
2. Update `db/sqlx-test-schema.sql` if the change references a new column.
3. Push → the `rust-sqlx-validation` CI workflow runs.
4. If it passes → download the new artifact → commit updated `.sqlx/`.
5. If it fails → fix the missing table/column in the schema file first.

**Do not hand-edit `.sqlx/` JSON files.** They are generated output.

## Railway deployment standard

Railway NIXPACKS builds the production binary. The build environment does not
have access to the Supabase database. After `.sqlx/` is committed:

1. Add `SQLX_OFFLINE=true` to the Railway environment variables for the
   `vantro-automation` service.
2. The Rust service binary will build without any database connection.
3. The binary connects to the database only at **runtime**, not at build time.

`SQLX_OFFLINE=true` must be set in Railway **before** enabling the Rust
service to accept traffic. If it is missing, the NIXPACKS build will fail
with `error: set SQLX_OFFLINE=true to use offline mode`.

## Maintenance checklist

Run through this checklist whenever `queries.rs` is modified:

- [ ] Does the query reference a table present in `db/sqlx-test-schema.sql`?
- [ ] Does the query reference a column present in that table?
- [ ] Are the Rust field types consistent with the Postgres column types?
      (e.g. `NUMERIC` → `f64` via `::float8`, `INTEGER` → `i32`/`i64`)
- [ ] Did `cargo sqlx prepare` succeed in CI?
- [ ] Was the updated `.sqlx/` artifact downloaded and committed?

## FAQ

**Q: Can I run `cargo sqlx prepare` locally?**

On Linux/macOS with Rust + Postgres installed:
```bash
export DATABASE_URL=postgres://postgres:postgres@localhost:5432/vantro_sqlx_test
psql "$DATABASE_URL" -f db/sqlx-test-schema.sql
cargo sqlx prepare --workspace -- --features server
```

On Windows without a linker: the `--features server` build requires a C
linker. Use WSL or let CI generate the cache for you.

**Q: What if a query uses a Supabase-specific type (e.g. `auth.uid()`)?**

`auth.uid()` is a Supabase function and is not available in vanilla Postgres.
Keep Supabase-specific logic in the Node backend. Rust queries must only use
standard Postgres features so they can be validated in the CI ephemeral DB.

**Q: The CI schema is different from production -- can the queries break?**

`cargo sqlx prepare` validates *types*, not data. As long as the column types
in `db/sqlx-test-schema.sql` match the production column types, the generated
`.sqlx/` cache is valid for the production database. Always keep the types in
sync. If a production column is `NUMERIC(14,2)` and the test schema has
`NUMERIC`, that is fine -- sqlx resolves both as `Decimal` / `f64`.

**Q: Is .sqlx/ committed for every developer?**

Yes. Once committed, every developer and CI job uses `SQLX_OFFLINE=true` for
the server feature build. No one needs a local database to build the binary.
Only the person refreshing queries needs to run `cargo sqlx prepare` (or
trigger the CI workflow).
